# 기획서 고도화 계획

## 현황

`product-specification.md`는 현재 **L0 (개요)** 수준:
- 기능 나열 + 1줄 설명
- 모델 속성 테이블
- 플로우는 단순 화살표 다이어그램

## 목표 깊이

| 레벨 | 설명 | 예시 |
|---|---|---|
| **L0** (현재) | 기능 목록 + 1줄 설명 | "반복 할일 건너뛰기 (다음 1회 / 특정 날짜까지)" |
| **L1** | 동작 상세 + 조건 분기 + 엣지 케이스 | 건너뛰기 시 turn 계산 로직, count 종료와 만났을 때의 동작, 시간 없는 할일에서 건너뛰기 시도 시 처리 |
| **L2** | 화면별 전체 시나리오 + 상태 전이 다이어그램 + 유효성 검증 규칙 | 이벤트 생성 화면의 필드별 입력 제약, 저장 버튼 활성화 조건, 모드별 UI 차이, 에러 처리 |

## 페이즈 구성

현재 19개 섹션을 **기능 도메인** 기준으로 7개 페이즈로 묶는다.
각 페이즈는 독립 실행 가능하며, 1회에 코드를 다시 탐색 → 해당 섹션만 L1~L2로 재작성한다.

---

### Phase 1: 이벤트 핵심 — 할일 + 일정 + 반복

**대상 섹션**: 3(이벤트 모델), 4(반복 이벤트), 16(미완료 할일 정책)

**현재 부족한 점**:
- 할일 생성/수정 시 유효성 검증 규칙이 없음
- 완료 처리 시 반복 할일의 turn 업데이트 상세 로직 없음
- 일정 수정 시 `.onlyThisTime` / `.fromNow`의 구체적 데이터 변환 과정 없음
- 반복 옵션별 파라미터 제약 조건 (interval 범위 등)이 표로만 있고 동작 설명 없음
- EventTime 3가지 형태의 overlap 판정, 타임존 보정 로직 없음
- 건너뛰기와 count 종료가 겹치는 엣지 케이스 없음

**L1 구체화 내용**:
- 할일/일정 생성 파라미터 유효성 규칙 (필수값, 조합 제약)
- 할일 완료/되돌리기의 전체 상태 전이 (반복 있을 때/없을 때/시간 있을 때/없을 때 매트릭스)
- 일정 반복 수정 범위별 구체적 데이터 변환 시나리오
- 반복 옵션별 다음 시간 계산 예시 (EveryWeek: 월수금, interval=2일 때 등)
- turn 규칙의 전체 생명주기 (생성 → 완료 → 건너뛰기 → count 종료)
- EventTime overlap 판정 로직 상세
- 미완료 할일의 자동 추가/제거 트리거 전체 목록

---

### Phase 2: 화면 — 캘린더 + 이벤트 상세 + 완료 목록

**대상 섹션**: 2(화면 구성)

**현재 부족한 점**:
- 캘린더 그리드의 이벤트 표시 규칙 (몇 개까지, 정렬 순서, 색상 우선순위) 없음
- 이벤트 상세 화면의 모드별 UI 차이 (어떤 필드가 보이고/숨겨지는지) 없음
- 저장 버튼 활성화 조건, 필드 간 연동 (하루종일 토글 시 시간 필드 변화 등) 없음
- 이벤트 목록의 정렬/그룹핑 규칙 없음
- 화면 전환 시 데이터 전달 규칙 없음
- 에러 상태 표시 (네트워크, 유효성) 없음

**L1 구체화 내용**:
- 캘린더 그리드: 이벤트 색상 바 표시 규칙, 최대 표시 수, "+N" 계산, 정렬 기준
- 일별 이벤트 목록: 섹션 구성, 이벤트 정렬 규칙 (시간순, 하루종일 우선 등), 빈 상태 처리
- 이벤트 상세: 모드별(생성/할일수정/일정수정/공휴일/구글/완료) 필드 매트릭스, 활성/비활성/숨김
- 이벤트 상세: 필드 간 연동 규칙 (하루종일↔시간, 반복↔시간 필수화 등)
- 이벤트 상세: 저장/수정 유효성 검증 + 에러 메시지
- 완료 할일 목록: 그룹핑 규칙 상세 (오늘/어제/이번달/월별/연별 경계), 페이지네이션 파라미터
- 화면 간 데이터 전달: Listener/Interactor 콜백 시점과 전달 데이터

### Phase 3: 태그 + 강조 이벤트 + 알림

**대상 섹션**: 5(이벤트 태그), 6(강조 이벤트), 7(알림)

**현재 부족한 점**:
- 태그 삭제 시 "관련 이벤트 일괄 삭제"의 구체적 범위 (완료 할일도? 이벤트 상세도?) 없음
- 태그 색상 결정의 우선순위 (이벤트에 태그+구글 colorId 둘 다 있을 때) 없음
- 강조 이벤트가 삭제/완료되었을 때의 처리 없음
- 알림 fire date 계산의 구체적 예시 없음
- 반복 이벤트의 알림 생성 범위와 갱신 타이밍 없음
- 알림 ID 관리 정책 (이벤트 삭제 시, 앱 재설치 시) 없음

**L1 구체화 내용**:
- 태그 CRUD 전체 시나리오 (생성 시 중복 체크, 삭제 시 cascade 범위)
- 태그 보이기/숨기기가 캘린더·이벤트 목록·위젯에 미치는 영향 매트릭스
- 구글 캘린더 태그와 커스텀 태그의 상호작용
- 강조 이벤트 지정/해제의 전체 상태 전이 + 엣지 케이스 (삭제, 완료, 반복 이벤트)
- 알림 시간 옵션 → fire date 변환 공식 (시간 이벤트 / 하루종일 / 타임존별)
- 알림 갱신 트리거 전체 목록 (이벤트 변경, 시간 변경, 반복 변경, 삭제 등)
- 로컬 알림 제한 (iOS 64개 제한) 대응 전략

---

### Phase 4: 구글 캘린더 연동

**대상 섹션**: 8(구글 캘린더 연동)

**현재 부족한 점**:
- OAuth 인증 상세 (스코프, 토큰 갱신 정책, 만료 처리) 없음
- 다중 계정 시 데이터 격리/합산 방식 없음
- 이벤트 동기화 주기와 캐시 무효화 정책 없음
- 반복 이벤트(rrule) 파싱과 앱 내 반복 모델 매핑 없음
- 에러 처리 (토큰 만료, API 제한, 네트워크 오류) 없음
- DB 참조 카운팅의 구체적 동작 없음

**L1 구체화 내용**:
- OAuth 플로우 상세 (스코프 목록, 토큰 저장, 갱신 전략, 만료 시 재인증)
- 다중 계정 데이터 아키텍처 (계정별 DB 분리, 합산 조회, ID 충돌 방지)
- Pool 패턴 상세 (DB Connection Pool, Repository Pool, Remote Pool)
- 이벤트 로딩 플로우 (API 호출 → 파싱 → 로컬 캐시 → UI 전파)
- rrule → 앱 반복 모델 변환 규칙
- 캐시 무효화 트리거와 갱신 전략
- 에러/재시도 정책

---

### Phase 5: 위젯

**대상 섹션**: 9(위젯)

**현재 부족한 점**:
- 각 위젯의 지원 사이즈 (small/medium/large/extraLarge) 없음
- 위젯별 표시 데이터의 구체적 구성 없음
- Timeline 갱신 정책 (after date, interval) 없음
- 위젯 Configuration Intent 파라미터 없음
- 위젯 ↔ 앱 데이터 공유 메커니즘 상세 없음
- 딥링크 URL 포맷과 파라미터 없음

**L1 구체화 내용**:
- 위젯별 사이즈 매트릭스 + 사이즈별 표시 내용 차이
- 각 위젯의 데이터 소스와 쿼리 범위
- Timeline Provider 갱신 정책 (시간 기반, 이벤트 기반)
- Intent Configuration 파라미터 (필터 옵션, 표시 옵션)
- TodoToggleIntent 동작 상세 (위젯 캐시 업데이트, 앱 DB 업데이트, 서버 동기화)
- 딥링크 URL 스펙 (호스트, 경로, 쿼리 파라미터)
- App Group 데이터 공유 방식 (UserDefaults vs 파일 vs SQLite 직접 접근)

### Phase 6: 계정 + 동기화

**대상 섹션**: 10(계정 & 인증), 11(동기화), 18(DB 마이그레이션)

**현재 부족한 점**:
- 로그인/로그아웃 시 UseCase Factory 전환의 구체적 영향 범위 없음
- 동기화 충돌 해결 정책 (같은 이벤트를 로컬+서버에서 동시 수정 시) 없음
- 오프라인 큐의 재시도 전략 (지수 백오프 등) 없음
- 마이그레이션 실패 시 복구 전략 없음
- DB 마이그레이션 이력 (v1→v6 각 버전별 변경 내용) 없음
- 백그라운드 동기화의 시스템 제약 (배터리, 네트워크, 빈도) 없음

**L1 구체화 내용**:
- 로그인 상태 전환 시 전체 영향 범위 (Repository 교체, Store 초기화, 위젯 갱신 등)
- 동기화 데이터 타입별 응답 구조 (created/updated/deletedIds)
- 충돌 해결 전략 (서버 우선? 최신 우선? 병합?)
- 오프라인 큐: 큐잉 단위, 재시도 간격, 최대 재시도, 실패 시 처리
- 백그라운드 동기화: iOS BGTaskScheduler 제약, 실행 보장 전략
- DB 마이그레이션 버전별 changelog

### Phase 7: 설정 + 기타

**대상 섹션**: 12(설정), 13(SharedDataStore), 14(딥링크), 15(피드백), 17(D-Day), 19(외부 의존성)

**현재 부족한 점**:
- 각 설정의 기본값이 대부분 "—"로 비어있음
- 설정 변경의 실시간 반영 범위 (어떤 화면이 즉시 갱신되는지) 없음
- SharedDataStore의 스레드 안전성, 메모리 관리 없음
- 딥링크 URL 포맷 전체 스펙 없음

**L1 구체화 내용**:
- 각 설정 항목의 정확한 기본값 (코드에서 추출)
- 설정 변경 → 영향받는 화면 매트릭스
- SharedDataStore 동시성 모델
- 딥링크 URL 전체 스펙 (호스트, 경로, 쿼리 파라미터, 예시)
- 피드백 전송 API 스펙
- D-Day 계산 공식 (타임존 처리 포함)

## 실행 방식

각 페이즈마다:
1. 해당 도메인의 코드를 **재탐색** (Usecase, ViewModel, Repository, Tests)
2. 기존 섹션을 **인라인 확장** (별도 파일 분리 X, 같은 문서 내에서 깊이 추가)
3. 추가 내용: 조건 분기표, 상태 전이 다이어그램, 구체적 예시, 엣지 케이스, 에러 처리
4. 완료 후 PR 또는 커밋

## 우선순위 추천

1. **Phase 1** (이벤트 핵심) — 앱의 핵심 도메인, 가장 복잡한 비즈니스 로직
2. **Phase 2** (화면) — 사용자 경험의 실체
3. **Phase 3** (태그+알림) — 이벤트와 밀접하게 연결
4. **Phase 4** (구글 캘린더) — 독립적이지만 복잡
5. **Phase 5** (위젯) — 독립적, 18종으로 분량 많음
6. **Phase 6** (계정+동기화) — 인프라 성격
7. **Phase 7** (설정+기타) — 비교적 단순
