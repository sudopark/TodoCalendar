## Project Overview

To-do Calendar is an iOS 17+ calendar app that combines schedule and todo management. Built with SwiftUI/UIKit hybrid, Tuist v3 for project generation, Combine for reactive data flow, and Swift 6.0.

## Setup

```bash
git checkout develop
./install/install.sh          # copies dummy config files into place
tuist fetch                   # resolve Swift Package dependencies
tuist generate --no-open      # generate TodoCalendar.xcworkspace
open TodoCalendar.xcworkspace
```

The `install.sh` script copies required secrets/config stubs from `install/` into `Tuist/ProjectDescriptionHelpers/` and `TodoCalendarApp/Resources/`. Without this step the project will not build.

The cloned app runs in offline mode only ‚Äî login and account features require additional server configuration.

## Build & Test

All schemes are auto-generated by Tuist. Run tests from Xcode (`Cmd+U`) or via xcodebuild:

```bash
# Run all tests for a specific module
xcodebuild test \
  -workspace TodoCalendar.xcworkspace \
  -scheme Domain \
  -destination 'platform=iOS Simulator,name=iPhone 16,OS=18.0'

# Run a specific test class
xcodebuild test \
  -workspace TodoCalendar.xcworkspace \
  -scheme Domain \
  -only-testing:DomainTests/CalendarSettingUsecaseImpleTests \
  -destination 'platform=iOS Simulator,name=iPhone 16,OS=18.0'
```

Key test schemes: `Domain`, `Repository`, `CalendarScenes`, `EventDetailScene`, `EventListScenes`, `SettingScene`, `MemberScenes`.

After changing `Project.swift` files or adding/removing sources, re-run `tuist generate` before building.

## Adding a New Scene (Xcode Template)

Install the Xcode file template once:
```bash
cd Template && sudo swift install.swift
```

Then in Xcode ‚Üí New File ‚Üí choose `Todo-Calendar-Scene`, enter the scene name. This generates `XXXScene+Builder.swift`, `XXXViewController.swift`, `XXXViewModel.swift`, `XXXBuilderImple.swift`, `XXXRouter.swift`, and optionally `XXXView.swift` (SwiftUI).

## Architecture

### Module Layout

```
Domain/          ‚Äì Models, Repository protocols, Usecase implementations
Repository/      ‚Äì Local (SQLite) and Remote (Alamofire) repository implementations
Presentations/
  Scenes/              ‚Äì Shared Scene/Builder protocols and UsecaseFactory protocol
  CommonPresentation/  ‚Äì Shared UI components and ViewAppearance
  CalendarScenes/      ‚Äì Calendar and day-event list screens
  EventDetailScene/    ‚Äì Event create/edit/detail screens
  EventListScenes/     ‚Äì Done-todo list and other standalone event lists
  MemberScenes/        ‚Äì Login and account screens
  SettingScene/        ‚Äì Settings screens
Supports/
  Extensions/          ‚Äì Swift extensions
  Common3rdParty/      ‚Äì Shared 3rd-party wrappers
  UnitTestHelpKit/     ‚Äì BaseTestCase, PublisherWaitable helpers
  TestDoubles/         ‚Äì Stub repositories and mock usecases
TodoCalendarApp/ ‚Äì App target: AppDelegate, SceneDelegate, Root/, Main/, Factories/
```

Dependency direction: `TodoCalendarApp ‚Üí Presentations ‚Üí Scenes/CommonPresentation ‚Üí Domain ‚Üê Repository`.

Presentation modules reference other presentation modules only through the `Scenes` framework's shared protocols ‚Äî never by importing each other directly.

### Scene Structure (MVVM + Router + Builder)

Every screen is a **Scene** built from these collaborating objects:

| File | Role |
|---|---|
| `XXXScene+Builder.swift` | Defines `XXXScene` protocol (inherits `Scene`) and `XXXSceneInteractor` protocol; also defines `XXXSceneBuilder` protocol |
| `XXXViewController.swift` | UIKit VC (or `UIHostingController` for SwiftUI); owns ViewModel; exposes `interactor = viewModel` |
| `XXXViewModel.swift` | Protocol + `XXXViewModelImple`; holds a `Subject` struct of `CurrentValueSubject`/`PassthroughSubject`; exposes `AnyPublisher` for view binding; delegates navigation to Router |
| `XXXRouter.swift` | Extends `BaseRouterImple`; implements `XXXRouting`; holds builder references for child scenes |
| `XXXBuilderImple.swift` | Wires ViewModel ‚Üí Router ‚Üí ViewController; injects `UsecaseFactory` and `ViewAppearance` |
| `XXXView.swift` *(SwiftUI only)* | `XXXViewState` (`@Published` + `bind(viewModel)`), `XXXViewEventHandler` (closures + `bind(viewModel)`), `XXXView` (reads from `@EnvironmentObject`), `XXXContainerView` (hosts state/handlers, used as `UIHostingController.rootView`) |

### Inter-Scene Communication

**Usecase (shared state):** Most cross-scene data sharing goes through `Usecase` objects backed by a singleton `SharedDataStore`. Different VMs that use the same Usecase type share state through that store without holding direct references to each other.

**Interactor (parent ‚Üí child):** When a parent scene owns a child scene (e.g., embedded in the same screen), the parent's Router calls `attachChild()` which returns the child's `SceneInteractor`. The parent VM stores this reference and calls methods on it directly.

**Listener (child ‚Üí parent):** When navigating forward (present/push), the current VM passes `self` as a `NextSceneListener` to the next VM via the Router. The next VM holds a weak listener reference and calls it to send messages back.

### UsecaseFactory

`Scenes` framework defines the `UsecaseFactory` protocol. `TodoCalendarApp/Factories/` contains two implementations ‚Äî `NonLoginUsecaseFactoryImple` and `LoginUsecaseFactoryImple` ‚Äî which are swapped at login/logout time. All Presentation `BuilderImple` objects receive a `UsecaseFactory` and call it to instantiate usecases.

### State Management in ViewModels

```swift
// Internal state
private struct Subject {
    let currentMonth = CurrentValueSubject<CalendarMonth?, Never>(nil)
}
private let subject = Subject()

// Exposed to view
var currentMonth: AnyPublisher<CalendarMonth, Never> {
    subject.currentMonth.compactMap { $0 }.eraseToAnyPublisher()
}
```

## Widget Extension

The widget target is `TodoCalendarAppWidget`, located at `TodoCalendarApp/AppExtensions/Widget/`.

### Widget Types (18 total)

Registered via `@main TodoCalendarWidgetBundle`, split into three sub-bundles:

| Bundle | Widgets |
|---|---|
| `BaseWidgetBundle` | TodayAndNext, Month, EventList, Today, ForemostEvent, NextEvent, NextRemainEvent |
| `ComposedWidgetBundle` | DoubleMonth, EventAndMonth, EventAndForemost, TodayAndMonth |
| `WeeksWidgetBundle` | OneWeek, TwoWeek, ThreeWeek, FourWeek, CurrentMonth, LastMonth, NextMonth |

Most use `StaticConfiguration`; `EventListWidget` and composed widgets that show a filtered event list use `AppIntentConfiguration` with `EventListComponentSelectIntent`.

### Widget Architecture Pattern

Each widget follows a consistent three-layer structure:

```
TimelineProvider
  ‚îî‚îÄ calls WidgetViewModelProviderBuilder
        ‚îî‚îÄ creates ViewModelProvider (uses WidgetUsecaseFactory)
              ‚îî‚îÄ produces ResultTimelineEntry<XxxWidgetViewModel>
                    ‚îî‚îÄ SwiftUI Widget View renders it
```

`ResultTimelineEntry<T>` is the common `TimelineEntry` type across all widgets ‚Äî it wraps a `Result<T, WidgetErrorModel>` so every widget view handles success/failure uniformly. On failure the view renders a shared `FailView`.

Timeline refresh uses a `nextUpdateTime` extension on `Date` ‚Äî it schedules the next reload at either 1 hour from now or midnight, whichever is sooner.

### Data Access from Widget

Widgets access data through the same repositories as the main app, but only locally:

- **SQLite** ‚Äì read-only access to the main app's database via `SQLiteService` (path: `AppEnvironment.dbFilePath(for: userId)`)
- **App Group UserDefaults** ‚Äì suite `"group.sudo.park.todo-calendar"` for calendar/appearance settings
- **Keychain** ‚Äì shared access group for auth credentials

The setup entry point is `AppExtensionBase`, which creates all repositories and hands them to `WidgetUsecaseFactory`. This mirrors the main app's `ApplicationBase` / `UsecaseFactory` pattern.

**Caching:** `CalendarEventsFetchCacheStore` and `HolidaysFetchCacheStore` are `actor`-based stores that cache fetched data within a single timeline refresh cycle. Cache can be selectively invalidated via `EnvironmentKeys.needCheckResetWidgetCache` / `needCheckResetCurrentTodo` in the shared UserDefaults.

### Intents & Configuration

`EventTypeSelectIntent` / `EventListComponentSelectIntent` implement `WidgetConfigurationIntent` (AppIntents framework). The user picks which event types (custom tags, Google Calendar calendars) to show. `EventTypeEntity` is the `AppEntity` used in the query ‚Äî loaded by `EventTypeQuery`, which fetches all custom tags and external calendars, filtering out holiday calendars.

`TodoToggleIntent` is an interactive `AppIntent` that lets users complete todos directly from the widget. On execution it toggles the todo, then calls `WidgetCenter.shared.reloadTimelines(ofKind:)` ‚Äî selectively for EventList/ForemostEvent/NextEvent widgets, or a full reload for foremost-marked todos.

### Deep Linking

`EventDeepLinkBuilder` produces `todocalendar://` URLs attached to widget taps:

```
todocalendar://calendar/event/todo?event_id=...
todocalendar://calendar/event/schedule?event_id=...&start=...&end=...
todocalendar://calendar/event/holiday?event_id=...
todocalendar://calendar/event/google?event_id=...&calendar_id=...
todocalendar://calendar?select=YYYY_MM_DD
```

### Widget Tests

Widget tests follow the same `BaseTestCase` + `PublisherWaitable` pattern as the main app. Test targets live at `Widget/Tests/` with `Doubles/` (stub repositories), `Usecases/`, and `ViewModelProviders/` subdirectories.

## Testing

Two test frameworks are used in this project:

1. **XCTest** ‚Äî subclass `BaseTestCase` from `UnitTestHelpKit`
2. **Swift Testing** ‚Äî use `@Test` / `#expect`, conform to `PublisherWaitable` directly (no `XCTestCase`)

`PublisherWaitable` (from `UnitTestHelpKit`) provides helpers for collecting Combine publisher emissions and works with both frameworks. It is not required for every test ‚Äî only when verifying publisher output.

See [`docs/testing.md`](./docs/testing.md) for full examples and API reference.
